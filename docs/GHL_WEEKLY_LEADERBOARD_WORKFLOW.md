# GHL Weekly Leaderboard Workflow

Use this for the Realist weekly leaderboard email in GoHighLevel AI Workflow Builder.

## Goal

Receive the weekly leaderboard webhook from Realist, update the contact, and send the email using the HTML payload generated by the app.

The Realist app now queues a `weekly_leaderboard_digest` webhook payload with:

- `subjectLine`
- `previewText`
- `emailBody`
- `emailHtml`
- `ctaUrl`
- `ctaLabel`
- `weeklyRank`
- `weeklyDeals`
- `weeklyTopAnalysts`
- `platformTotalDeals`
- `platformAvgCapRate`
- `platformAvgCashOnCash`
- `platformAvgDscr`
- `insight`
- `unsubscribeUrl`
- `ghlTags`

## Trigger

Build a workflow in GHL that starts from the inbound webhook endpoint used by `GHL_WEBHOOK_URL`.

Recommended trigger filter:

- `sendEmail` equals `true`
- `eventType` equals `weekly_leaderboard_digest`

If your webhook workflow handles other Realist events too, branch immediately on `eventType`.

## Contact Update Mapping

In the webhook workflow, map these fields onto the contact before sending:

- Email: `email`
- First name: `firstName`
- Tag: `weekly-leaderboard`
- Tag: `weekly-digest`
- Optional custom field `realist_last_event_type`: `eventType`
- Optional custom field `realist_weekly_rank`: `weeklyRank`
- Optional custom field `realist_weekly_deals`: `weeklyDeals`
- Optional custom field `realist_platform_total_deals`: `platformTotalDeals`
- Optional custom field `realist_last_weekly_digest_at`: `eventTs`

If you already use one Realist source tag, keep that too:

- Tag: `realist-user`

## AI Workflow Builder Structure

Recommended structure:

1. Trigger: inbound webhook
2. If/Else: `eventType == weekly_leaderboard_digest`
3. Find or create contact from `email`
4. Update contact fields and apply tags
5. If/Else: `emailHtml` exists and is not empty
6. Send email using HTML body
7. Else branch: send plain-text fallback using `emailBody`
8. Optional: write note/activity to contact with `subjectLine` and `eventTs`

## Send Email Step

This is the important part.

Do not ask the AI builder to rewrite the email from the JSON. Do not feed it the ranking data and let it generate copy. The Realist app already generated the exact email.

The send step should use the webhook payload directly:

- Subject: `{{subjectLine}}`
- Preview text / preheader: `{{previewText}}`
- Recipient: contact email
- From name: `Daniel Foch`
- From email: `daniel@realist.ca` or your approved sending domain in GHL
- Reply-to: `daniel@realist.ca`

For body content:

- Preferred: set the email body to raw/custom HTML using `{{emailHtml}}`
- Fallback: if GHL refuses raw HTML variables in that builder path, use a custom code/html block that outputs `{{emailHtml}}` without escaping
- Last resort: use `{{emailBody}}` as plain text

## What To Tell The GHL AI Builder

Use this prompt inside the GHL AI workflow builder:

```text
Build a workflow for Realist weekly leaderboard emails.

Trigger from inbound webhook data.
Only continue when eventType equals weekly_leaderboard_digest and sendEmail equals true.

Find or create the contact by email.
Update the contact first name from firstName.
Apply tags realist-user, weekly-digest, and weekly-leaderboard.
Store weeklyRank, weeklyDeals, eventTs, and platformTotalDeals into custom fields if available.

Then send an email to the contact.
Use subjectLine as the subject.
Use previewText as the preheader.
Use emailHtml as the raw HTML body exactly as provided.
Do not rewrite, summarize, or regenerate the email copy.
If emailHtml is missing, send emailBody as the fallback plain-text body.
Use Daniel Foch as sender name and daniel@realist.ca as reply-to.
```

## If GHL Escapes The HTML

If the email arrives showing HTML tags, the workflow is escaping `emailHtml` instead of rendering it.

Fix:

- Replace the normal text block with a raw HTML/custom HTML block
- Insert the `emailHtml` variable directly into that block
- Disable AI rewriting or content enhancement on that step
- Disable template wrappers that inject their own layout around the payload

Do not paste `{{emailHtml}}` into a rich text paragraph block if GHL converts it to text.

## Branching Logic

Add these guards:

- If `email` is empty: stop
- If `subjectLine` is empty: stop
- If both `emailHtml` and `emailBody` are empty: stop
- If `eventType` is not `weekly_leaderboard_digest`: route to your generic Realist workflow branch or stop

## Suggested Custom Fields

These are optional but useful in GHL:

- `realist_weekly_rank`
- `realist_weekly_deals`
- `realist_last_event_type`
- `realist_last_weekly_digest_at`
- `realist_platform_total_deals`

## Testing

Before enabling live sends:

1. Trigger `/api/admin/weekly-digest/send` from Realist admin.
2. Confirm a `weekly_leaderboard_digest` webhook hits GHL.
3. Confirm the contact is found by email.
4. Confirm the email subject uses `subjectLine`.
5. Confirm the message renders as formatted HTML, not visible tags.
6. Confirm the unsubscribe link lands on `https://realist.ca/api/email/unsubscribe?...`.

## Known Constraint

The app side is ready to send rich HTML through the webhook. The final rendering quality now depends on GHL using `emailHtml` as raw HTML in the email step.

If the AI builder cannot do that cleanly in your account, the fallback is:

- store `emailHtml` in a custom field
- use a custom HTML email template step that injects that field directly
