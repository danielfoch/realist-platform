# US Listings Nightly Sync — Design Spec

**Status:** Design only. No scheduler code yet.
**Owner:** Platform / Listings.
**Anchors:** `server/routes.ts` (`/api/ingest/us-listings`, `/api/ingest/us-listings/reconcile`), `shared/schema.ts` (`usListings`, `usListingPriceHistory`), `client/src/pages/UsListings.tsx` (freshness badges).

This spec describes how the nightly refresh of US listings should behave end-to-end. It assumes the ingest path patched in this session (deterministic `lastSeenAt`/`lastCheckedAt`, append-only price history) is the only writer. The actual scheduler, refresh button, and admin UI are explicitly out of scope here — see §10.

---

## 1. Market Cadence

**Atomic unit:** `(source, city, state)` — e.g. `(homeharvest, Houston, TX)`. Every market is scraped, ingested, and reconciled as one indivisible operation. Cross-market work is parallel but independent.

**Default cadence per market:** once every 24h, with jitter to avoid thundering-herd at midnight.

**Tiering** (drives next-run scheduling, not write semantics):

| Tier | Criterion | Target interval | Rationale |
|---|---|---|---|
| Hot | ≥1 saved listing belongs to the market, OR ≥10 active listings touched by a saved search in the last 7d | 6h | Investor attention concentrated here |
| Warm | ≥500 active listings, no saved-listing pressure | 24h | Default working set |
| Cold | <500 active listings or no user touch in 30d | 72h | Keep fresh but cheap |

Tiering is computed at scheduler start, not baked into the table. A market can change tier between runs.

**Scrape window:** UTC 04:00–10:00 (overnight in all NA timezones). Outside the window the scheduler is idle.

**Concurrency cap:** at most N=4 markets scraping in parallel per source. HomeHarvest is the only source today; this prevents IP-rate-limit cascades.

## 2. Local HomeHarvest Source-of-Truth Flow

HomeHarvest runs locally on Clyde's box, not on the Replit server. The flow:

```
[scheduler]
   │  (per market, in priority order)
   ▼
[HomeHarvest CLI] ──► CSV/JSON for (city, state)
   │
   ▼
[normalizer]                      ── maps HH fields to ingest payload schema
   │
   ▼
[POST /api/ingest/us-listings]    ── 25-row batches, x-ingest-token header
   │                                  Each batch is independently idempotent.
   ▼
[POST /api/ingest/us-listings/reconcile]
   │                                  ONE call per market after all batches land.
   │                                  Marks anything not in activeSourceIds as
   │                                  off_market_unknown.
   ▼
[scheduler] ── records market_sync_runs row, computes next run
```

**Source-of-truth rules:**
1. HomeHarvest's *snapshot* of a market at scrape time is authoritative for "what is currently for sale." Anything in our DB for that `(source, city, state)` not in the snapshot's `activeSourceIds` is presumed gone.
2. Per-listing fields (price, beds, baths, status, etc.) are authoritative *only when the listing appears in the current snapshot*. Listings absent from the snapshot keep their last-known values frozen; we only flip `isActive=false` and stamp `offMarketDetectedAt`.
3. Historical truth (price history, first-seen date) is owned by us, never by the source. HomeHarvest does not get to rewrite the past.

**Why batch + reconcile, not one giant atomic swap:** lets a market partially succeed. If the scrape collects 4,800 of 5,000 listings and dies on the last page, the 4,800 we got are still upserted and visible; only the missing 200 are penalized by reconcile (which won't run, since the scrape failed — see §6).

## 3. Ingest Idempotency

Idempotency is required at three layers because nightly retries will replay batches.

**Layer 1 — Listing upsert.** Already implemented: `ON CONFLICT (source, source_id) DO UPDATE`. Replaying the same batch is a no-op on every field except `lastCheckedAt`/`updatedAt`, which advance to "now" by design (that's the whole point of `lastCheckedAt`).

**Layer 2 — Price history append.** Already implemented: partial unique index on `(listing_id, scraped_at, new_price) WHERE scraped_at IS NOT NULL` + `ON CONFLICT DO NOTHING`. Replaying a batch with the same `scrapedAt` collapses to one row per real price change.

**Layer 3 — Reconcile.** Reconcile is idempotent as written: it updates only rows where `isActive = true AND source_id NOT IN activeSourceIds`. Replaying with the same `activeSourceIds` set finds zero rows the second time. ⚠️ However, replaying with a *different* (smaller) `activeSourceIds` would incorrectly mark fresh listings inactive. **Scheduler contract: reconcile runs at most once per market per scrape run, only after all batches for that run have ack'd 200.**

**Run identity.** Each nightly scrape of a market gets a `run_id` (UUID, generated by scheduler, included in every batch's metadata). This is the unit of "did this complete." Used by:
- Reconcile gating (one reconcile per `run_id`).
- Observability (§8) — group all batches under one run.
- Failure handling (§6) — retry by `run_id`, not by individual batch.

## 4. Price-Change Detection

Already shipped end-to-end (`b12ad7660`). The nightly sync changes nothing about the mechanism; it only changes *cadence*.

**Recap of the contract:**
- `oldPrice` is the pre-upsert snapshot's `list_price`, taken before the same batch's upsert.
- `newPrice` is the validated payload row's `list_price`.
- A history row is written iff both are non-null and differ.
- Race protection: oldPrice is read from `existingByKey` (pre-fetched in the same batch), not a post-upsert re-read. Two concurrent writers cannot mis-attribute a transition to each other.
- Dedupe: `(listing_id, scraped_at, new_price)` partial unique index, `ON CONFLICT DO NOTHING`.

**What nightly adds:** the daily cadence means most listings see at most one price-change row per day. Burst editing on the source (a listing that drops 5 times in one hour) collapses to whichever value was current when our scrape ran. That's an acceptable loss for v1 — sources don't expose intraday price history reliably anyway.

**Read side:** `fetchPriceChangeStats` already computes `lastPriceChangeAt`, `lastPriceChangeAmount`, `lastPriceChangePercent`, `originalListPrice`, `priceCutCount` from the table. Nothing to add for nightly.

## 5. Freshness Semantics

Shipped this session. The nightly sync is *the* mechanism that makes the freshness badges meaningful.

| Field | Set on | Set to | Preserved on update? |
|---|---|---|---|
| `firstSeenAt` | First insert only | `payload.scrapedAt ?? now` | ✅ yes (omitted from UPDATE SET) |
| `lastSeenAt` | Every successful ingest where the listing is in the batch | `payload.scrapedAt ?? now` | ❌ bumped every time |
| `lastCheckedAt` | Every successful ingest where the listing is in the batch | `now` (server time) | ❌ bumped every time |
| `updatedAt` | Every row write | `now` (server time) | ❌ row write time |

**Badge thresholds** (already in `UsListings.tsx`):
- 🟢 `<24h` since `lastCheckedAt ?? lastSeenAt`
- 🟡 `1–7d`
- 🔴 `>7d` (truly stale — likely a market not being scraped, not a dead listing)

**Critical gap that nightly closes:** today, a listing is only `lastCheckedAt`-bumped when it appears in *some* batch. With ad-hoc ingest, a market that wasn't scraped for 3 days has every listing showing red even though it's not delisted. With nightly cadence, every active listing in every covered market is touched at least once per tier interval (6h/24h/72h), and the badges become a true freshness signal.

**Edge case:** listings that go off-market between scrapes. After reconcile flips them to `isActive=false, offMarketDetectedAt=now`, `lastCheckedAt` is also bumped to the scrape completion time. Their freshness anchor stays current, but `isActive=false` filters them out of the default surface. No badge confusion.

## 6. Failure / Retry Rules

Failure can happen at three levels. Each has its own retry contract.

**Batch-level failure** (one POST to `/api/ingest/us-listings` returns non-2xx):
- Retry the batch up to 3 times with exponential backoff (2s, 8s, 32s).
- Batches are independent — one failed batch does not block sibling batches in the same run.
- After 3 failures, mark the *run* as `partial` (not failed). The successful batches stay in the DB; reconcile is skipped (see below).

**Scrape-level failure** (HomeHarvest CLI errors out, returns no rows, or returns suspiciously few):
- Mark run as `failed`. **Do not call reconcile.** This is the single most important rule in the whole spec. Reconcile run with a half-scraped `activeSourceIds` would mark thousands of healthy listings off-market.
- Next run attempts the market again at next tier interval. No automatic immediate retry — failures are usually source-side rate limits or outages, and hammering makes it worse.
- After 3 consecutive failed runs of the same market, page Clyde (just a Slack message for now; alerting plumbing later).

**Reconcile-level failure:**
- Reconcile already requires `scrapeSucceeded=true && activeSourceIds.length > 0` (existing guard at `server/routes.ts:1759`). Don't relax this.
- If reconcile itself 500s, retry once after 60s. After that, mark run `partial` and move on — better to leave a few stale `isActive=true` rows for one cycle than risk double-killing them.

**Run state machine:**
```
pending → running → (succeeded | partial | failed)
                          │            │          │
                          │            │          └─ no reconcile, retry next cycle
                          │            └─ some batches landed, no reconcile
                          └─ all batches + reconcile ok
```

Terminal states are never overwritten; a re-run gets a new `run_id`.

## 7. Market Sync Status Table

New table: `us_listing_sync_runs`. Pure observability, no behavior depends on it being populated (scheduler must work if the table is dropped).

```
us_listing_sync_runs
────────────────────
id                uuid pk
run_id            uuid not null            -- scheduler-generated, shared across batches
source            text not null            -- 'homeharvest'
city              text not null
state             text not null
tier              text not null            -- 'hot' | 'warm' | 'cold'
started_at        timestamptz not null
completed_at     timestamptz
status            text not null            -- 'pending' | 'running' | 'succeeded'
                                           -- | 'partial' | 'failed'
listings_seen     int                      -- payload.activeSourceIds.length
listings_upserted int                      -- sum of batch upserted counts
listings_reconciled_off int                -- reconcile.staleRows.length
batches_attempted int
batches_succeeded int
error             text                     -- last error message if failed
created_at        timestamptz default now()

INDEX (source, city, state, started_at DESC)  -- "latest run per market" lookups
INDEX (status, started_at DESC)               -- admin dashboard filters
```

**Not in this table:**
- Per-listing change events (they live in `us_listing_price_history`).
- Per-batch detail (logged to stdout; structured log aggregation is later work).
- Saved-listing impact metrics (computed on-the-fly from `usListings` + `savedListings` joins).

**Retention:** 90 days. Older rows roll off via nightly cleanup. Aggregate metrics (runs/day, success rate) are computed live, not pre-aggregated.

## 8. Observability / Admin View

A single read-only page: `/admin/listings/sync` (gated by existing admin auth). Powered by a new endpoint `GET /api/admin/us-listings/sync-status`.

**Page layout:**

1. **Top strip — 24h health KPIs.**
   - Runs attempted, succeeded, partial, failed.
   - Total listings touched, total price changes detected.
   - Worst-tier age: max(now - latestSucceededRun) across all markets.

2. **Markets table.** One row per `(source, city, state)`, sorted by `latestSucceededRun ASC` (most-stale first):
   - Market | Tier | Listings (active/total) | Last run status | Last success age | Last 7 runs (sparkline of status)
   - Click row → drawer with last 20 runs for that market, full timeline.

3. **Recent failures feed.** Last 50 `failed` or `partial` runs across all markets, with error message.

4. **Reconcile audit.** Last 100 reconciles, with `listings_reconciled_off` count. Anomaly detection: any reconcile where >10% of the market was marked off-market gets highlighted yellow — that's the signature of a half-scraped run that slipped past the guard.

**Endpoints needed (build in this order):**
1. `GET /api/admin/us-listings/sync-status` — markets table + 24h KPIs (single round-trip).
2. `GET /api/admin/us-listings/sync-status/runs?city=&state=` — drawer detail.
3. `GET /api/admin/us-listings/sync-status/failures?limit=50` — failures feed.

All three are read-only from `us_listing_sync_runs` + `usListings` aggregates. No new writes.

**What the user sees outside admin:** unchanged. Freshness badges on `/listings/us` are already the user-facing observability surface.

## 9. Saved-Listing Refresh Priority

Saved listings (rows in `saved_listings` / `saved_searches` that reference a `usListings.id`) get preferential cadence. Two mechanisms:

**A. Tier elevation (preferred, already in §1).** Any market containing ≥1 saved listing or matching ≥10 hits/7d of a saved search is promoted to Hot tier (6h cadence). This catches the common case without per-listing logic.

**B. Targeted refresh queue (only if A is insufficient).** A small `saved_listing_refresh_queue` table holds individual `listing_id`s that need an immediate re-check (e.g., the user just saved it, or it hasn't been seen in 36h despite its market being scraped). The nightly scheduler drains this queue *before* starting market scrapes, hitting HomeHarvest with a single-listing lookup per queue entry.

**Decision: build A only for v1.** B adds a second write path and a new queue table; A reuses the market-level flow we already trust. Revisit B if user complaints show market-level cadence still leaves saved listings stale.

**One non-negotiable rule either way:** saved listings *never* get reconciled off-market by a partial scrape. The reconcile guard at `server/routes.ts:1759` already requires `scrapeSucceeded=true`, which prevents this for the market-level path. If we add path B, single-listing fetches must follow the same rule — a failed lookup keeps the row as-is, never flips `isActive=false`.

## 10. What Not to Build Yet

These are deliberately deferred. Spec changes if/when we add them; nothing in v1 should anticipate them.

| Deferred | Why deferred | When to revisit |
|---|---|---|
| User-facing **refresh button** | Cadence + tiering should cover 95% of perceived staleness. Manual refresh invites abuse and source-rate-limit blowback. | After 2 weeks of nightly running, if users still ask for it. |
| Per-listing **change notifications** (email/push) | `queueUsListingChangeNotifications` already collects the change set in ingest; the *delivery* layer is separate work. | After admin view shows price changes flowing reliably. |
| **Intraday** price tracking | Sources don't expose it; would require polling at high cost for marginal value. | Never, unless a paid source is added. |
| **Multi-source merging** (HomeHarvest + Realtor.com + Zillow on the same listing) | One source today. Schema supports it via `(source, source_id)`, but the dedupe logic across sources is non-trivial. | When a second source is contracted. |
| **Backfill of historical price snapshots** | Price-history table is forward-only by design. Backfilling would require trusting source history, which we don't. | Never. |
| **Reconcile across multiple runs** (e.g., "haven't seen this listing in 3 runs → off-market") | Current single-run reconcile is sufficient because HomeHarvest snapshots are complete per market. | If we add a partial-snapshot source. |
| **Automated tier rebalancing** based on user behavior over time | Static thresholds in §1 are fine for v1. | After 1 month of usage data. |
| **Pause/resume controls** for individual markets | YAGNI. Killing the scheduler process is sufficient. | If a market needs to be quarantined for >24h. |
| **Saved-listing targeted refresh queue** (§9 path B) | Tier elevation should suffice. | Only if path A proves inadequate. |
| **The scheduler itself** | This spec is the contract. Implementation is the next task, not part of this one. | Next task. |

## Open Questions for Implementation Task

1. Where does the scheduler run? Replit cron, an external box (Clyde's), or a small worker the Express server spawns? Affects how `run_id` is minted and how failures are surfaced.
2. Does `us_listing_sync_runs` get written by the scheduler directly (needs DB credentials there) or via a new `/api/ingest/us-listings/run-status` endpoint? Latter keeps the DB single-writer.
3. Slack alerting plumbing — does Realist already have a generic admin alert webhook, or do we add one for this?

None of these block this spec; they're decisions for the implementation kickoff.
