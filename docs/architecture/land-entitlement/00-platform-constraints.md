1. PostgreSQL/Drizzle is the system of record; IDs are UUID-like `varchar` values generated by `gen_random_uuid()` (`shared/schema.ts:1-25`).
2. The production schema declares a PostGIS geometry passthrough; spatial work belongs in PostgreSQL/PostGIS rather than a second GIS store (`shared/schema.ts:7-18`).
3. Scheduled jobs use `node-cron`; the monthly winner job imports it at `server/monthlyWinnerEmail.ts:1` and registers a monthly schedule near the bottom of that file.
4. Report pages use cards, badges and Recharts; `client/src/pages/BuildingPermitsReport.tsx:1-20` shows the shared component/chart pattern.
5. `client/src/pages/MultiplexUnderwriterPage.tsx:1-10` states the address-first dual-takeout purpose; its result types and UI distinguish MLI hold from condo termination.
6. The same page's `ProvenanceBadge` component (near `client/src/pages/MultiplexUnderwriterPage.tsx:120`) exposes `verified`, `inferred`, `assumption`, and `estimate` labels.
7. `client/src/pages/MarketReportBuilder.tsx:1-30` uses the shared query client and Recharts, providing the saved-report/chart composition pattern.
8. `src/api-routes.ts:1-30` shows Express route construction, typed requests and a database adapter boundary.
