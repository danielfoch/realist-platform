# Realist.ca UX/UI Audit — July 2026

**Date:** 2026-07-02
**Method:** 10 parallel code-audit agents (one per surface) over the repo at commit `ee801bf`, judged against best-in-class comparators (Zillow, Redfin, HouseSigma, BiggerPockets, Linear, Stripe, Airbnb). A live-site sweep did not run (rate limit), so findings are code-based; production may differ from `main`.
**Overall verdict:** ~4/10 vs. best-in-class. The gap is not missing features — it is reachability, trust, and consolidation. The best work (AI underwriter, agent API, premium gating, native events) is hidden or dead code; the roughest work (placeholder people, sample-data demos, empty leaderboards, broken links) is what visitors see.

## Surface scores

| Surface | Score | Surface | Score |
|---|---|---|---|
| First-touch / homepage | 4.5 | Community & network | 4 |
| Navigation & IA | 4 | Conversion & monetization | 3 |
| Tools suite | 4.5 | Visual consistency | 5 |
| Map / listings | 4.5 | Mobile | 4.5 |
| Content / insights | 4.5 | AI story vs. ambition | 3 |

## Strategy decisions (founder, 2026-07-02)

1. **Audience:** investors first, with a visible "For Professionals" lane (nav item + homepage band → Power Team / partner onboarding).
2. **Core product:** one "Analyze" suite — Yield Map → Analyzer → AI Underwriter (flagship), sharing one property context. Investor OS becomes "My Deals"; Listing Intelligence goes to Labs until real.
3. **AI bet:** on-site "Ask Realist" agent (`/api/ask` over the existing agentApi tools), MCP stays as distribution channel. Demo target: Sept 15, 2026.
4. **Monetization:** software stays free. The real monetization is **deal-through**: analyzed deals flow to network professionals via the CRM; realtors/mortgage brokers pay a 25% referral fee on successful deals for access to Realist leads; knowledge professionals (architects, planners) pay nothing. End-state: an AI investor realtor that sees deals through. The BRA-for-free-premium tier is deprecated pending legal review.

## Mobile experience (responsive web + Capacitor iOS/Android shell)
### [critical] Google sign-in will almost certainly fail inside the iOS/Android app (disallowed_useragent)
GoogleSignInButton.tsx:16-20 does window.location.href = '/api/auth/google/start', which redirects the Capacitor WKWebView/Android WebView itself to accounts.google.com - mobile/capacitor.config.ts:18 even whitelists accounts.google.com in allowNavigation, confirming the OAuth dance is meant to happen inside the webview. Google has blocked OAuth from embedded webviews since 2021 (403 disallowed_useragent; WKWebView UA lacks the Safari/Version tokens, Android WebView UA contains '; wv)'). mobile/SHIP-CHECKLIST.md explicitly plans to verify 'RSVP to a meetup in-app while logged out (account-creation flow)' - that flow dead-ends at Google's error page. Auth is the front door of the app; if it breaks, tickets, saved deals, and the CRM are all inaccessible in-app.
FIX: Detect Capacitor.isNativePlatform() in GoogleSignInButton and route the OAuth flow through the system browser instead: open /api/auth/google/start via @capacitor/browser (already a dependency in mobile/package.json), complete OAuth in Safari/Chrome Custom Tab, and deep-link back via @capacitor/app URL handler (ca.realist.app:// or universal link) that exchanges a one-time code for the session cookie. Test before store submission - this is a hard blocker for the Sept 15 demo if the demo is in-app.

### [critical] The flagship yield map toolbar overflows/clips on phones - the exact surface every mobile homepage search lands on
CapRates.tsx:3793-3851 renders the search row as <div className="flex gap-2 items-center"> with NO flex-wrap: a flex-1 search input (intrinsic min-width ~170px from the input's default size attribute; shadcn Input has w-full but no min-w-0) plus 'Find Deals' (px-4 + icon), 'Save Search', an X clear button, and 'Filters' - roughly 270-300px of non-shrinkable buttons before the input. On a 375px viewport that is ~460px of content in a 375px row inside a page whose root is overflow-hidden (line 3763), so the input gets crushed to unusable width and/or the Filters/Save buttons clip off-screen. This is the page the /discover NL command bar (MapHomepage.tsx:326) and the 'Open the Yield Map' CTAs route every query to, and CapRates.tsx has essentially no small-screen treatment: 0 md: classes, 2 sm:, 9 lg: across 4,386 lines - docs/DISTRIBUTION_ARCHITECTURE.md itself calls it 'desktop-first map UI'. Compare Zillow/Redfin mobile where search is a full-width bar and filters live behind a single icon.
FIX: Restructure the toolbar for <lg: full-width search input on its own row with the Find Deals action inside it (icon button), and collapse Save Search/Clear/Filters into a single filter icon-button opening a bottom sheet. Also hide the two-line 'Deal sourcing' header paragraph (lines 3776-3778) on mobile - it burns ~90px of a ~660px viewport above the map.

### [major] Deal analyzer on mobile is a blind 20+ field form: the live Quick Preview only exists on desktop
In Home.tsx:1315 the analyzer is grid lg:grid-cols-3 - on desktop the sticky Quick Preview sidebar (line 1374, sticky top-24) shows live cash flow/yield as you adjust sliders, which is the product's core delight. Below lg everything stacks single-column, so on a phone the preview card ends up BELOW the entire DealInputs form (~20 fields across 4 cards plus a Collapsible advanced section, DealInputs.tsx:355-517) and below the Calculate button - users adjust assumptions with zero feedback, then scroll to discover results. In the embedded map-sheet variant it is worse: Home.tsx:1172 gates results behind showResults && leadCaptured, so a phone user in the map workspace fills the long form and then hits a lead-capture modal before seeing a single number. Best-in-class mobile calculators (Zillow affordability, BiggerPockets) keep a persistent result readout on screen.
FIX: On <lg viewports render a slim fixed bottom bar (safe-area padded) showing Est. Cash Flow + Cap Rate that live-updates as inputs change, with a 'Full results' expand. Reconsider the lead-capture gate on mobile - show headline metrics free and gate save/export/PDF instead (the copy at Home.tsx:1412 already implies that model).

### [major] Precision financial inputs are slider-only with 20px thumbs - unusable for exact underwriting on touch
PercentInput (DealInputs.tsx:76-113) renders ONLY a Radix slider - no numeric field - and it is used for interest rate (0-15 range, 0.1 steps = 150 stops across a ~300px track, ~2px per stop on a phone), down payment, vacancy, rent growth, maintenance, capex, management, appreciation. The thumb is h-5 w-5 = 20px (client/src/components/ui/slider.tsx:21), less than half Apple's 44px minimum. An investor who knows their quote is 4.29% physically cannot enter it on a phone; they get whatever their thumb lands on. For a platform whose brand is underwriting rigor, imprecise inputs undermine trust in every downstream number. Number inputs also use type="number" without inputMode="decimal" (DealInputs.tsx:65,136), so some Android keyboards show the full keyboard instead of a keypad.
FIX: Pair every PercentInput slider with a small editable numeric field (pattern already exists in the codebase for interest rate label at line 392 - extend it to accept typed values), enlarge the thumb to h-6 w-6 minimum with a larger invisible hit area, and add inputMode="decimal" to CurrencyInput/NumberInput.

### [major] The Capacitor app is a pure webview of the desktop-oriented site with zero app-worthy adaptations - and its own docs say the mobile gap is UX, not packaging
Nothing in client/src adapts when running natively: the only isNativePlatform() consumers are push registration and hiding GetAppBanner. So the 'app' is the website with a hamburger menu - no bottom tab bar (the standard for Zillow/Redfin/BiggerPockets), no native share, no offline state (@capacitor/network is installed in mobile/package.json but literally nothing in client/src imports or uses it - launch with no connectivity shows a blank webview error), and useIsMobile() (client/src/hooks/use-mobile.tsx) is only consumed by the unused shadcn sidebar. docs/DISTRIBUTION_ARCHITECTURE.md decision #4 already concedes: 'The real mobile gap is UX, not packaging... Mobile v1 should be a small, mobile-first My Realist surface'. mobile/README.md:119 acknowledges 'Pure passthrough wrappers can get rejected' (App Store guideline 4.2). Shipping the current wrap risks rejection AND a 2-star first impression from users who expect an app, not a website in a frame.
FIX: Before store submission, add a minimal native-mode layer keyed off isNativePlatform(): a 4-tab bottom nav (Discover / Analyzer / Saved / Events), an offline banner using @capacitor/network, and make 'My Realist' (saved analyses, alerts) the app's home surface per the distribution doc. This doubles as the 4.2 review answer.

### [major] Geolocation is requested on page load without a user gesture on the web - and is silently dead inside the native apps
GeolocateOnMount (MapHomepage.tsx:45-65, also used in CapRates.tsx:4272) calls navigator.geolocation.getCurrentPosition on mount. On /discover this fires a browser location-permission prompt just to pan a blurred, non-interactive, pointer-events-none decorative background map - a classic trust-burning anti-pattern (Airbnb/Zillow only ask on 'near me' actions), and once a user hits Block, the real map on /tools/cap-rates can never geolocate. Inside the Capacitor apps it is worse: mobile/ios/App/App/Info.plist has no NSLocationWhenInUseUsageDescription and mobile/android/app/src/main/AndroidManifest.xml has only INTERNET (line 34), so webview geolocation cannot even prompt - 'near me' discovery is silently broken in the shipped app.
FIX: Remove GeolocateOnMount from the decorative /discover background entirely (default to TORONTO_CENTER). On the real map, trigger geolocation only from a 'Near me' button tap. For the apps, add NSLocationWhenInUseUsageDescription + ACCESS_COARSE_LOCATION and wire Capacitor's geolocation permission bridge before submission.

## AI story: how AI actually shows up in the product vs. the "best AI realtor in the world" ambition (hero claims, AI-labeled tools, LLM call sites, MCP/agent infrastructure)
### [critical] The AI flywheel the homepage sells is literally broken: two duplicate, incompatible /api/ai/defaults implementations collide
server/aiDefaults.ts and server/aiMarketDefaults.ts BOTH register GET /api/ai/defaults, POST /api/ai/train, and GET /api/ai/training-stats. aiDefaults.ts registers first (server/routes.ts:765 via index.ts:600, before index.ts:601 registers the other), so it wins. But the client (client/src/components/DealInputs.tsx:310-330) expects the aiMarketDefaults.ts shape ({found, metrics: {median,...}, sampleCount}) while the winning route returns {success, defaults: {value,...}} with no 'found' key — so `data?.found` is always falsy and the 'AI defaults available — learned from N analyses' banner (DealInputs.tsx:452-455) can never render. The two modules also CREATE TABLE ai_market_defaults with incompatible columns (value/sample_size vs median_value/sample_count), so the nightly training cron (scheduleNightlyTraining from aiMarketDefaults) writes against a schema that may not exist. Net effect: the homepage's core AI promises — 'Every deal teaches Realist your buy box' (InvestorStart.tsx:313) and 'The real estate AI that gets smarter with every deal you analyze' (MapHomepage.tsx:503) — are non-functional in the shipped UI.
FIX: Delete one implementation (keep aiDefaults.ts's more robust trainer), align its response shape to what DealInputs.tsx expects (or update the client type), keep a single table schema with a migration, and add an integration test that asserts the analyzer prefill banner renders when priors exist. Then surface /api/ai/training-stats on the homepage as real social proof ('priors trained on N analyses across M markets').

### [critical] The only real LLM feature in the product is orphaned — zero inbound links
server/multiplexReportWriter.ts is the ONLY Anthropic SDK call site in the entire codebase (verified by grep across client/src and server/). Its front end, /tools/multiplex-underwriter, appears nowhere except its route registration in App.tsx:193 — it is absent from the ToolsHub tool grid (client/src/pages/ToolsHub.tsx lists 17 other tools), absent from Navigation.tsx, and absent from both homepages. Likewise /account/api-keys — the front door to the MCP/agent story — is linked from nowhere in the client (Navigation.tsx has no account menu entry). The two most differentiated AI assets the company owns are unreachable by navigation; a user could use realist.ca for a year and never encounter actual AI.
FIX: Add 'AI Multiplex Underwriter' to ToolsHub with a featured/AI badge and cross-link it from the deal analyzer results and Will-It-Plex/Multiplex Feasibility pages (the reverse link already exists at MultiplexUnderwriterPage.tsx:455). Add 'Connect Claude / API keys' to the account/user menu and a 'Use Realist inside Claude' marketing section on the homepage or /premium.

### [critical] 'AI' is a cosmetic label on deterministic calculators at every first-touch surface, which trains users to distrust the brand
InvestorStart.tsx:158 hero says 'Canadian real estate investing, underwritten by AI' and the 'Use AI underwriting' badge (line 229) routes to /tools/analyzer — a standard calculator with no model behind it (its one AI element, the smart-defaults banner, is broken per issue 1). The /discover NL command bar (MapHomepage.tsx:304-373) presents as conversational AI search but just forwards the raw string to /tools/cap-rates?q=..., where parseNaturalLanguageQuery (server/routes.ts:6342) is regex matching: city substring, under/over price, beds. A query like 'cash-flowing duplex near a hospital with a legal basement suite' silently degrades to an unfiltered city search with no feedback about what was understood. The 'AI Search Brief' card (Home.tsx:1231) merely echoes the query string back. Against Zillow's actual NL search or ChatGPT-class expectations, users immediately feel the gap between the claim and the behavior — fatal positioning for a company whose stated goal is being THE best AI realtor.
FIX: Either make the claims true or narrow them. Fastest path to true: route the command bar through an LLM parse (the Anthropic SDK is already a dependency) or at minimum echo the parsed filters back ('Understood: Hamilton · under $900k · 4+ units — 2 filters ignored') so failure is visible and correctable. Rewrite hero badges so 'AI' appears only where a model or learned prior actually runs.

### [major] The AI-benchmark trust surfaces are broken and contradictory: two benchmark brands, one 404, one 'Coming Soon', one unrouted page with illustrative scores framed as independent
ReportsHub.tsx:54 and MarketReport.tsx:437 link to /insights/market-report/homebench-ai-realtor-benchmark, but App.tsx registers no route for that path (only /insights/market-report exact) — the promoted card 404s. The fully built HomeBenchReport.tsx is therefore dead code; its SEO meta calls it 'An independent benchmark of frontier AI models' while its fine print (line 939) admits 'Scores shown are an illustrative public run' over hardcoded constants, and the methodology section promotes a third-party product (meetyourhomies.com 'Homies harness') — brand confusion and a credibility risk if it ever ships as-is. Meanwhile a second, differently-named benchmark ('RealBench AI Realtor Benchmark', RealBenchReport.tsx) is routed at /reports/realbench-ai-realtor-benchmark and is a permanent 'Coming Soon' shell. For a company whose thesis is AI credibility, its public AI-measurement artifacts are a 404, a stub, and placeholder data.
FIX: Pick ONE benchmark brand. Register the route (or remove the ReportsHub/MarketReport cards until real), replace hardcoded scores with actual run data or clearly watermark every chart 'illustrative', and resolve the Homies relationship (co-brand deliberately or remove the promotion).

### [major] ListingIntelligence sits in main navigation as an 'AI-ready' product but is a hardcoded demo sandbox
Navigation.tsx:41 lists 'Listing Intelligence' with an 'AI-ready' badge as a first-class nav destination, but the page (client/src/pages/ListingIntelligence.tsx) runs entirely on sampleListingIntelligence, sampleProfessionals, and sampleListingFeedbackEvents from @shared/listingIntelligence — fabricated professional profiles with ratings, demo feedback that mutates only local state, and a 'Local demo data only. A prospect clicking a nav item lands in a mock and reasonably concludes the whole AI story is vaporware — contaminating trust in the parts that are real (the underwriter, the MCP server).

### [major] No conversational surface on realist.ca despite the agent infrastructure already existing — the 'AI realtor' only exists inside Claude Desktop
Grep confirms zero chat/assistant UI in client/src. Yet server/agentApi.ts already exposes exactly the tool set an on-site agent needs (underwrite listing/custom, NL find-deals, saved analyses, market reports, rates) with auth, rate limiting, and usage metering — it just requires users to leave the site, mint an API key from an unlinked page, and configure Claude Desktop. The highest-leverage AI-native move available is an on-site 'Ask Realist' agent: a Claude tool-use loop over the same 8 agent endpoints, surfaced as (a) the /discover command bar's real backend and (b) a listing/analyzer-level Q&A panel ('Does this pencil at 20% down?', 'What kills this deal?'), governed by the multiplexReportWriter doctrine (math computes, model narrates, numbers validated). This converts already-built infrastructure into the visible product the hero copy is promising, and it is what would actually differentiate against Zillow/Redfin, whose AI is search-summary-deep only.
FIX: Build a server-side /api/ask endpoint: Claude + tool definitions that call the existing agentApi handlers internally, stream the answer with provenance badges reused from MultiplexUnderwriterPage. Ship it first on the deal analyzer results page where user intent and computed context are richest; make the discover command bar submit to it.
