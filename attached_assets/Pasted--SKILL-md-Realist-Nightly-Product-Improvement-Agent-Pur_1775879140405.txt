# SKILL.md — Realist Nightly Product Improvement Agent

## Purpose

This skill runs a nightly autonomous product-improvement cycle for **Realist.ca** inside the Replit project at:

`https://replit.com/@danielfoch/Realist-Platform`

The mission is to make Realist into the best-in-class real estate intelligence platform for **investor members** and **realtor members**, with a strong focus on:

* seamless user experience
* powerful deal analysis
* richer underwriting data capture
* DDF-native workflow intelligence
* AI inference on deal quality and investor intent
* lead generation and routing between investors and realtors
* enforceable referral and membership rails
* compounding product quality through small nightly shipped improvements

This agent should work like a focused product engineer + growth operator + UX optimizer.

It should make high-confidence improvements directly in code, test them, commit them, and log what changed.

---

## Core Product Vision

Realist is not just a listing site.
It should become a **real estate Bloomberg terminal + underwriting engine + lead marketplace** for investors and realtors.

### Realist should do all of the following:

#### 1) Serve investor members extremely well

* Help investors find, filter, and analyze deals quickly
* Make underwriting easy, fast, and habit-forming
* Store prior analyses so users build a persistent investing workflow
* Let users compare properties, neighborhoods, and strategies over time
* Capture structured data about what users analyze, what assumptions they use, what makes them convert, and what they come back to
* Eventually build AI inference around user preferences and deal fit

#### 2) Serve realtor members extremely well

* Help realtors connect with investor leads
* Let realtors present themselves clearly to investor users
* Capture realtor preferences, specialties, markets, asset types, and lead routing rules
* Ask realtors what referral fee they are willing to pay for platform-generated leads
* Build systems that turn traffic and underwriting activity into qualified introductions

#### 3) Make deal analysis world-class

* Powerful deal analyzer
* Rich assumptions and scenario modeling
* Ability to analyze listings in-platform
* Ability to analyze user-sourced addresses or opportunities
* Track underwriting inputs and outputs
* Build a memory layer around investor behavior and assumptions
* Use DDF/listing context wherever possible
* Over time, infer what each investor considers a “good deal”

#### 4) Use DDF intelligently

* Realist should not just display listings; it should become an intelligence layer on top of DDF data
* The system should capture what users browse, save, analyze, revisit, share, or inquire on
* Use those actions to improve recommendations, routing, and lead quality
* Treat DDF behavior as underwriting intent data

#### 5) Build lead and transaction infrastructure

* Connect investor members to realtor members
* Route based on market, product type, strategy, and seriousness
* Make referral economics explicit
* Require referral agreements where appropriate
* Require user agreements during signup or onboarding
* Create a system where Realist becomes a serious lead-gen and referral platform, not just a content/tool layer

## Strategic Product Themes

The agent should continuously improve the site along these themes.

### A. User Experience

Prioritize:

* fewer clicks
* clearer onboarding
* less friction to first value
* faster path to analyzing a deal
* better sign-up conversion
* better member segmentation
* less confusion between realtor and investor journeys

The ideal experience:

* user lands on site
* immediately understands what Realist does
* identifies as investor or realtor
* signs up with minimal friction
* sees tools/features relevant to that role
* quickly gets value
* deeper data capture happens naturally as they use the platform

### B. Dual-Sided Marketplace Design

Realist has at least two key user types:

* investors
* realtors

The site should make this obvious without feeling cluttered.

The platform should gradually evolve toward:

* investor dashboard / deal workflow
* realtor dashboard / lead + profile + referral workflow
* lead matching layer between the two

### C. Data Moat

Every improvement should consider whether it helps Realist capture:

* underwriting assumptions
* user preferences
* search behavior
* save behavior
* inquiry behavior
* geography and asset type preferences
* return frequency
* lead quality signals
* referral economics

The product should compound knowledge, not just pageviews.

### D. AI Inference Layer

The long-term goal is not just calculators.
The long-term goal is inference.

Examples:

* Which properties fit this investor’s actual behavior?
* What cap rate / cash flow / renovation / neighborhood profiles does this user consistently respond to?
* What kind of realtor should this investor be matched with?
* Which lead is high intent?
* Which listing deserves a prompt follow-up or suggested action?

Nightly improvements should move the codebase toward capturing the data needed for that future.

## Non-Negotiable Product Requirements

The agent should prioritize or preserve the following requirements.

### 1. Role-aware onboarding

During signup or early onboarding, clearly identify whether the user is:

* an investor
* a realtor
* both

### 2. Referral agreements

Both realtor users and investor users should be steered into appropriate referral or platform agreements as part of onboarding/signup when legally and operationally appropriate.

The product should move toward explicit agreement capture, stored acceptance, timestamping, and easy retrieval.

### 3. Realtor referral fee capture

For realtor members, the system should ask what referral fee they are willing to pay for leads generated through Realist.

This should be stored as structured data, not just free text.

### 4. Analysis memory

The site should preserve user underwriting activity where possible:

* properties analyzed
* assumptions entered
* outputs generated
* edits/revisions
* favorites/saves
* inquiries/submissions

### 5. DDF-connected behavior tracking

Track meaningful interactions with DDF-sourced property pages and tools.

### 6. Lead routing logic

The site should increasingly support structured routing of opportunities from investor users to realtor members based on:

* geography
* asset class
* strategy
* stage of intent
* referral economics

## Operating Philosophy For This Nightly Agent

### Bias toward shipping

This is a one-hour nightly cycle, not a six-month architecture exercise.

The agent should:

* make one meaningful improvement or a few tightly related improvements
* avoid sprawling rewrites unless clearly justified
* prefer shipping compounding improvements over big speculative rebuilds

### Preserve quality

Every change should aim to improve one or more of:

* clarity
* speed
* conversion
* trust
* data capture
* deal analysis depth
* marketplace usefulness

### Keep the product coherent

Do not add random features.
Do not add visual clutter.
Do not create “AI” gimmicks that do not improve user outcomes.

### Make the vision clearer over time

A major objective is clearer communication of what Realist is and why it matters.

The homepage and onboarding should increasingly communicate:

* what problem it solves
* for whom
* why it is better
* what users can do immediately
* what realtors get
* what investors get

## Runbook

The nightly agent should execute in this order.

### Step 1: Enter repo and sync

* Open the Replit project
* Pull latest code / inspect current state
* Review recent commits and project notes if available
* Avoid clobbering recent work

### Step 2: Inspect product state

Quickly review:

* homepage
* signup / auth flow
* onboarding
* investor flow
* realtor flow
* deal analyzer
* property/listing flow
* profile/account settings
* analytics / event capture code
* agreement/referral logic if present

### Step 3: Choose one focused objective

Select one scoped improvement for this run.
Examples:

* add realtor referral fee field to onboarding
* add investor/realtor role segmentation to signup
* improve hero copy and CTA structure
* save underwriting sessions to user profile
* instrument analysis events
* add agreement acceptance checkbox + persistence

### Step 4: Implement

* Modify code directly
* Keep changes cohesive
* Avoid touching unrelated systems unless required
* Prefer backwards-compatible changes

### Step 5: Test

At minimum:

* run app
* validate changed flows
* inspect console / logs
* confirm no obvious regressions

### Step 6: Commit and log

* write a clear git commit message
* push to GitHub if configured
* create a short run log summarizing:

  * what was changed
  * why it matters
  * what files were touched
  * what should come next

```bash

DATE=$(date +%F)

cd "$WORKDIR"

# Pull latest work if needed

  git pull --rebase || true
fi

# Invoke your OpenClaw agent here with the SKILL.md loaded.
# Replace the command below with your actual runner.
openclaw run \
  --workspace "$WORKDIR" \
  --skill skills/realist_nightly/SKILL.md \
  --task "Run the Realist nightly improvement cycle. Spend up to 1 hour shipping one high-leverage improvement, test it, commit it, and log it." \
  | tee "$LOGDIR/$DATE.log"
```
