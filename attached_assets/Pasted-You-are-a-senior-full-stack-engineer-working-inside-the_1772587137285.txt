C) CALCULATION ENGINE (CORE LOGIC)
Implement a shared library module:
- /lib/mortgage/amortization.ts (or similar)

Requirements:
1) Fixed mortgage:
   - Rate constant for selected term and for the full horizon if user assumes renewal at same rate OR (better) treat fixed as “5-year fixed then renew at same fixed” for 10y/25y comparisons.
   - Implement it as: for 10y/25y, assume fixed renews at the SAME fixed rate unless you add a renewal slider. Keep it simple, document assumption.

2) Variable mortgage:
   - Interest rate changes over time per scenario path (monthly steps).
   - Payment type assumption:
     - Default: “Adjustable-rate mortgage (ARM-style)” where payment recalculates when rate changes to keep amortization on track.
     - Optionally support “VRM constant payment” if you already do it, but default to adjustable for clarity.

3) Output metrics per horizon:
   For each horizon (5y, 10y, 25y):
   - totalPaid
   - totalInterestPaid
   - totalPrincipalPaid
   - endingBalance
   - table rows by year: Year, Starting Balance, Payment, Interest Paid (year), Principal Paid (year), Ending Balance, Avg Rate (year)

4) Scenario engine for variable:
   - Represent rate path as an array of monthly rates for at least 25*12 months.
   - Base case path:
     - If “market base case” exists in DB, use that array (or yearly nodes interpolated to monthly).
     - If missing, base case = constant variable rate.
   - Best/Worst:
     - If direction = rising: worst = +severity, best = -severity (or 0)
     - If direction = falling: best = -severity, worst = +severity
     - Apply severity over first 5 years primarily; for 10/25, keep rate at the final level after year 5 (simple) unless you have stored longer forecasts.

5) Sliders mapping:
   - Severity in bps translates to max delta over 5 years.
   - Speed/front-load controls the curve. Implement a simple curve:
     - evenly distributed: linear delta from month 1..60
     - front-loaded: 60% of delta occurs in first 24 months
     - back-loaded: 60% occurs in last 24 months
   Make this deterministic and unit-testable.

D) 10Y BOND YIELD “MARKET BASE CASE” FORECAST
Update the rate scraper (or add a new job file) that:
1) Pulls Canada 10-year bond yield from your existing data source (if you already store it). If not stored:
   - Add a minimal fetch step from a reliable source already used in the app OR allow manual admin input (prefer: reuse existing macro data pipeline if present).
2) Produce a simple implied variable-rate path:
   - We are not building a true yield-curve model. Use a simple rule-based mapping:
     - Base variable rate today = current variable from DB
     - Expected change over next 5 years = k * (10y_yield - rolling_12m_avg_10y_yield)
       where k default ~0.5–0.8, clamp to +/- 200 bps
   - Convert expected change to a monthly path using the “even/front/back-loaded” profile (default evenly)
3) Store forecast in DB:
   - Table: rate_forecasts
     columns: id, created_at, forecast_type ('ca10y_simple'), term_months, path_json (array of monthly rates), assumptions_json
   - Also store the 10y yield value used and the avg used.

If you cannot reliably fetch 10y yield in this repo, then:
- implement the DB table + UI hooks,
- and default forecast_type to null with graceful UI fallback to constant-rate base case.
Do not block shipping the calculator on missing macro data.
